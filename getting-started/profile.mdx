## Phase 1: Registration & Onboarding

### 1. Localization & Language Selection

<Steps>
  <Step title="Launch the Application">
    Upon opening the app, users configure their preferred language.
    
    <Accordion title="Backend Security: Zero-Trust Handshake" icon="server">
      **What happens under the hood:** The moment the app launches, it initiates a secure "handshake" with the Sasini servers. We assume every connection is potentially hostile. The backend instantly scans the connection for SSL/TLS 1.3 encryption certificates to ensure the network hasn't been compromised before allowing the app to load.
    </Accordion>
  </Step>
  
  <Step title="Select Preferred Language">
    Users can choose between **English**, **Swahili**, **French**, or **German**.
    
    <Accordion title="Backend Security: Secure State Initialization" icon="server">
      **What happens under the hood:** When a language is selected, the backend generates a secure, localized environment state saved in an encrypted, tamper-proof local storage container. This ensures malicious actors cannot inject foreign scripts or XSS attacks through manipulated language files.
    </Accordion>
    
    <div align="center">
      <img src="/images/language-selection.jpeg" alt="Language Selection Screen" style={{ maxWidth: '300px', margin: '0 auto', borderRadius: '12px' }} />
    </div>
  </Step>
</Steps>

### 2. Farmer Validation Protocol

<Steps>
  <Step title="Input Farmer ID">
    The user inputs their unique Sasini Farmer ID into the system.

    <Accordion title="Backend Security: Parameterized Query Execution" icon="server">
      **What happens under the hood:** To absolutely prevent SQL Injection, the backend sanitizes the Farmer ID using parameterized queries. It strips the input of executable code before querying the encrypted database. Invalid IDs throw a generic `Validation Failed` error to give hackers zero clues about our structure.
    </Accordion>
  </Step>

  <Step title="System Verification">
    If valid, the system allows the farmer to proceed.

    <Accordion title="Backend Security: Anti-Brute Force Rate Limiting" icon="server">
      **What happens under the hood:** Hackers use automated "bots" to guess IDs. The Sasini backend employs strict **Rate Limiting**. If a device attempts too many invalid IDs, the backend's firewall permanently blacklists their IP address, stopping brute-force attacks instantly.
    </Accordion>
  </Step>
</Steps>

### 3. Account Creation & MFA Setup

<Steps>
  <Step title="Enter Credentials">
    The user inputs their **Email Address** and creates a secure **Password**.

    <Accordion title="Backend Security: TLS 1.3 Payload Encryption" icon="server">
      **What happens under the hood:** When the user taps "Submit", their password is NOT sent as plain text. It is encrypted locally on the phone. Even if intercepted over public Wi-Fi, it appears as an unbreakable string of randomized alphanumeric garbage.
    </Accordion>
  </Step>
  
  <Step title="Enforce Password Policies">
    The system strictly enforces complex password rules.

    <Accordion title="Backend Security: Cryptographic Password Hashing" icon="server">
      **What happens under the hood:** Sasini **never** stores actual passwords. The backend runs the password through a cryptographic hashing algorithm with a unique mathematical "salt". This turns the password into a one-way hash, making database breaches useless to attackers.
    </Accordion>
  </Step>
</Steps>

<Tip>
  **Best Practice:** We recommend farmers use a combination of words they can easily remember but computers cannot easily guess (e.g., `Green#Coffee!Harvest99`).
</Tip>

### 4. Email Link Verification

<Steps>
  <Step title="Dispatch Verification">
    A secure verification link is sent to the registered email address.

    <Accordion title="Backend Security: CSPRNG Token Generation" icon="server">
      **What happens under the hood:** The backend generates a Cryptographically Secure Pseudo-Random Number Generator (CSPRNG) token tied to the user's IP, device, and a strict timer. It is mathematically unguessable.
    </Accordion>
  </Step>
  
  <Step title="Action Required">
    The user clicks the secure link to confirm ownership.

    <Accordion title="Backend Security: Anti-Phishing Link Validation" icon="server">
      **What happens under the hood:** The system instantly invalidates the token after a single click, completely neutralizing "Replay Attacks" where a hacker tries to reuse an old link.
    </Accordion>
  </Step>
</Steps>

## 5. Profile Finalization & Dashboard

Once the email link is clicked and verified, the backend finalizes the registration. 

<Steps>
  <Step title="Finalize Registration">
    The user's secure profile is saved, their Unique Digital ID is permanently linked to their email, and the app automatically fetches their customized data. 

    <Accordion title="Backend Security: JWT Session Issuance" icon="server">
      **What happens under the hood:** Upon successful verification, the server issues a secure **JSON Web Token (JWT)**. This token acts as a highly secure digital passport that is required for every single action the user takes moving forward. The token rotates securely, meaning even if a session is hijacked, it becomes useless within minutes, automatically locking out unauthorized users.
    </Accordion>
  </Step>

  <Step title="Enter the Dashboard">
    The user is then safely navigated to their main **Dashboard**, ready to explore the app!

    <Accordion title="Backend Security: Immutable Audit Logging" icon="server">
      **What happens under the hood:** The final step of registration triggers an automated, immutable log in our deep backend. The system records the successful secure registration in a write-only database vault. This provides complete traceability and ensures total transparency and security monitoring for the lifetime of the farmer's account.
    </Accordion>
  </Step>
</Steps>

## Phase 2: Login & Access

### 1. Credential Submission & Risk Analysis

<Steps>
  <Step title="Enter Login Details">
    The user inputs their email and password.
    
    <Accordion title="Backend Security: Constant-Time Hash Comparison" icon="server">
      **What happens under the hood:** The backend retrieves the hashed password and applies the same algorithm to the newly typed password. To prevent "Timing Attacks," the server uses a constant-time comparison, taking the exact same microsecond to respond whether the password is right or wrong.
    </Accordion>
  </Step>
  <Step title="Evaluate Login Context">
    The system checks where the login is coming from.
    
    <Accordion title="Backend Security: Heuristic Threat Modeling" icon="server">
      **What happens under the hood:** The backend analyzes device fingerprint and IP location. If a farmer logs in from Nairobi, and 10 minutes later a login attempt occurs from Eastern Europe, the heuristic engine flags it as impossible and blocks the foreign IP.
    </Accordion>
  </Step>
</Steps>

### 2. Contextual Risk Analysis

Before granting access, the system evaluates the context of the login attempt to ensure it is the real user.

<Steps>
  <Step title="Evaluate Login Context">
    The system checks where the login is coming from.
    
    <Accordion title="Backend Security: Heuristic Threat Modeling" icon="server">
      **What happens under the hood:** The backend analyzes the login metadata. It checks the device fingerprint, the IP address location, and the login velocity. For example, if a farmer logs in successfully from Nairobi, Kenya, and then 10 minutes later a login attempt occurs from a server in Eastern Europe, the backend's heuristic engine instantly flags it as impossible, blocks the foreign IP, and temporarily locks the account.
    </Accordion>
  </Step>
  <Step title="Manage Failed Attempts">
    If the user types the wrong password, the system carefully manages the failure.
    
    <Accordion title="Backend Security: Exponential Backoff Protocol" icon="server">
      **What happens under the hood:** To defeat "Credential Stuffing" (where hackers use lists of stolen passwords from other websites), the backend uses Exponential Backoff. After 3 failed login attempts, the server introduces a time delay. After 5 attempts, the account is temporarily locked, requiring an email reset. 
    </Accordion>
  </Step>
</Steps>

### 3. Session Issuance & Dashboard Entry

Once the credentials and context are verified, the user is granted access to the system.

<Steps>
  <Step title="Issue Secure Session">
    The backend authorizes the device and generates a secure session.
    
    <Accordion title="Backend Security: Short-Lived JWT & Device Binding" icon="server">
      **What happens under the hood:** The server issues a highly secure JSON Web Token (JWT). This token acts as a digital, temporary passport required for every action the user takes. It is strictly bound to the user's specific device fingerprint. Furthermore, the token is "Short-Lived"—it automatically expires in the background and is silently refreshed. This means even if a highly sophisticated hacker managed to steal the session data, the token would be useless to them within minutes.
    </Accordion>
  </Step>
  <Step title="Access Granted">
    The user is routed securely to their personalized dashboard, and an immutable log of the login is recorded for audit purposes.
    
    <Accordion title="Backend Security: Immutable Access Logging" icon="server">
      **What happens under the hood:** The successful login triggers an automated log in our deep backend vault. This system records the time, IP, and device of the successful login in a write-only database. This provides complete traceability and ensures total transparency for the lifetime of the farmer's account, guaranteeing absolute data integrity.
    </Accordion>
  </Step>
</Steps>

## Phase 3: Lifecycle & Security Settings

To protect farmers who may share devices or occasionally forget their credentials, the Sasini App features a robust account recovery and lifecycle management suite.

### 1. Logging Out

Farmers can securely terminate their active session at any time from the settings menu.

<Accordion title="Backend Security: JWT Revocation & Blacklisting" icon="server">
  **What happens under the hood:** Clicking "Log Out" does not just clear the screen. The frontend deletes the secure token from local storage, while the backend actively adds that specific session token to a cryptographic "Blacklist." Even if a hacker managed to copy the token right before the user logged out, the server will reject it entirely.
</Accordion>

### 2. Forgot Password (Account Recovery)

If a user forgets their password, they can initiate a secure recovery pipeline.

<Steps>
  <Step title="Request Reset">
    The user clicks "Forgot Password" on the login screen and enters their email.
  </Step>
  <Step title="Secure Email Link">
    A temporary reset link is dispatched. It expires in exactly 15 minutes.
    
    <Accordion title="Backend Security: Anti-Enumeration & Signed URLs" icon="server">
      **What happens under the hood:** To prevent hackers from guessing which emails are registered, the backend will *always* say "If this email exists, a reset link has been sent," regardless of whether the email is valid. The link sent contains a securely signed URL payload that can only be decrypted by our servers.
    </Accordion>
  </Step>
  <Step title="Create New Password">
    The user clicks the link and establishes a new, strictly parameterized password.
  </Step>
</Steps>

### 3. The Strict "Change Password" Loop

If a user *knows* their password but wants to update it for security reasons, the Sasini App enforces a **Strict Re-authentication Loop**. This ensures that someone who finds an unlocked phone cannot hijack the account.

<Steps>
  <Step title="Initiate Change">
    The user clicks "Change Password" in their profile. To prove ownership, the app **immediately and deliberately logs the user out**.
  </Step>
  <Step title="Re-Authenticate">
    The user must log back in with their *current* password. 
  </Step>
  <Step title="The Password Dialog">
    Upon successful login, a special system dialog intercepts the dashboard and says: *"Identity Verified. Click Continue to change your password."*
    
    <div align="center">
      <img src="/images/change-password-dialog.jpeg" alt="Identity Verified Dialog" style={{ maxWidth: '300px', margin: '1rem auto', borderRadius: '12px' }} />
    </div>
  </Step>
  <Step title="Update & Auto-Logout">
    The user inputs their new password (which must pass all complexity parameters). Once submitted, the system **automatically logs them out again**.
    
    <div align="center">
      <img src="/images/change-password-page.jpeg" alt="Enter New Password Page" style={{ maxWidth: '300px', margin: '1rem auto', borderRadius: '12px' }} />
    </div>
  </Step>
  <Step title="Final Verification">
    The user must now log in with the *new* password, cementing it in their memory and verifying the database update was successful.
    
    <Accordion title="Backend Security: Cryptographic Salt Rotation" icon="server">
      **What happens under the hood:** When the password is changed, the backend doesn't just hash the new password; it generates a brand new cryptographic "Salt" (a random data string) and destroys the old one. It also instantly revokes all active sessions on any other devices, forcing anyone using the account to re-authenticate immediately.
    </Accordion>
  </Step>
</Steps>

### 4. Account Deactivation

We respect the privacy and autonomy of our farmers. Users can request to deactivate or permanently delete their accounts from the Security Settings.

<Warning>
  **Warning:** Account Deletion is permanent. Once the grace period expires, historical harvest data, payout ledgers, and AI insights cannot be recovered.
</Warning>

<Steps>
  <Step title="Initiate Deactivation">
    The user selects "Deactivate Account" and must undergo an MFA email verification to prove intent.
    
    <div align="center">
      <img src="/images/deactivate-account-dialog.jpeg" alt="Account Deactivation Dialog" style={{ maxWidth: '300px', margin: '1rem auto', borderRadius: '12px' }} />
    </div>
  </Step>
  <Step title="Grace Period">
    The account is placed in a suspended state for 30 days, during which the user can cancel the request by simply logging back in.
  </Step>
  <Step title="Permanent Purge">
    After 30 days, the data is wiped.
    
    <Accordion title="Backend Security: Cascading Data Anonymization" icon="server">
      **What happens under the hood:** To maintain the integrity of Sasini's overall agricultural and financial analytics while complying with data protection laws, the backend performs a "Cascading Anonymization." The user's personal identifiable information (PII) like Name, Email, and Phone Number are permanently deleted and replaced with randomized string values. Their harvest weights and dates remain for corporate accounting, but they can mathematically never be traced back to the individual farmer again.
    </Accordion>
  </Step>
</Steps>
